## 1. Основные принципы

### 1.1 Ясность и специфичность

```
❌ Плохо: "Проверь код"

✅ Хорошо: "Проверь функцию authenticate_user() на:
1. SQL injection уязвимости
2. Утечку credentials в логах
3. Корректность хеширования паролей"
```

### 1.2 Контекст имеет значение

```
❌ Плохо: "Как исправить ошибку?"

✅ Хорошо: "В FastAPI приложении при POST /api/users возникает
ValidationError: 'email field required'. Вот код эндпоинта:
[код]
И схема Pydantic:
[схема]
Как исправить?"
```

### 1.3 Формат вывода

```
✅ Явно указывай ожидаемый формат:

"Проанализируй код и верни результат в формате:
## Найденные проблемы
| # | Файл:строка | Проблема | Severity |
|---|-------------|----------|----------|

## Рекомендации
1. ...
2. ..."
```

## 2. Chain-of-Thought (CoT)

### 2.1 Zero-shot CoT

Добавь фразу **"Давай разберём пошагово"** или **"Let's think step by step"**.

```
Промпт:
"В этом коде есть баг. Давай разберём пошагово:
1. Что делает каждая строка
2. Какие есть edge cases
3. Где может быть ошибка

[код]"
```

### 2.2 Few-shot CoT (с примером)

```
Промпт:
"Найди уязвимости в коде.

Пример анализа:
Код: `query = f"SELECT * FROM users WHERE id = {user_id}"`
Анализ:
1. Используется f-string для SQL запроса
2. user_id вставляется напрямую без экранирования
3. Уязвимость: SQL Injection
4. Исправление: использовать параметризованные запросы

Теперь проанализируй:
[твой код]"
```

### 2.3 Когда использовать CoT

| Задача | Нужен CoT? |
|--------|------------|
| Дебаггинг сложной логики | Да |
| Простое форматирование | Нет |
| Архитектурные решения | Да |
| Поиск по файлам | Нет |
| Security review | Да |
| Рефакторинг | Да |

## 3. Few-Shot промптинг

### 3.1 Структура

```
[Инструкция]
[Пример 1: вход → выход]
[Пример 2: вход → выход]
[Твоя задача: вход → ?]
```

### 3.2 Пример для code review

```
Промпт:
"Оцени качество функции по шкале 1-5.

Пример 1:
```python
def add(a, b):
    return a + b
```
Оценка: 3/5
Причина: Работает, но нет типов, docstring, валидации.

Пример 2:
```python
def add(a: int, b: int) -> int:
    """Add two integers."""
    if not isinstance(a, int) or not isinstance(b, int):
        raise TypeError("Both arguments must be integers")
    return a + b
```
Оценка: 5/5
Причина: Типы, docstring, валидация, чистый код.

Теперь оцени:
[твой код]"
```

### 3.3 Количество примеров

| Сложность задачи | Примеров |
|------------------|----------|
| Простая | 1 |
| Средняя | 2-3 |
| Сложная | 3-5 |

## 4. Role Prompting

### 4.1 Базовый паттерн

```
"Ты — [роль] с [X лет опыта] в [области].
Твоя задача: [что делать].
Твой стиль: [как отвечать]."
```

### 4.2 Примеры ролей

#### Security Expert
```
"Ты — senior security engineer с 10+ лет опыта в OWASP,
penetration testing и secure code review.
Проанализируй код на уязвимости из OWASP Top 10.
Для каждой найденной проблемы укажи:
- CWE номер
- Severity (Critical/High/Medium/Low)
- Конкретный exploit сценарий
- Код исправления"
```

#### Code Architect
```
"Ты — software architect, специализирующийся на
микросервисной архитектуре и DDD.
Проанализируй структуру проекта и предложи улучшения.
Фокус на: separation of concerns, SOLID, testability."
```

#### Performance Engineer
```
"Ты — performance engineer, эксперт по оптимизации
Python/FastAPI приложений.
Найди bottlenecks в коде: N+1 queries,
неэффективные алгоритмы, memory leaks."
```

## 5. Task Decomposition

### 5.1 Разбивка сложной задачи

```
❌ Плохо:
"Сделай полный аудит проекта"

✅ Хорошо:
"Проведи аудит проекта в 4 этапа:

ЭТАП 1: Структура
- Найди дубликаты файлов
- Найди мусорные файлы
- Проверь организацию папок

ЭТАП 2: Безопасность
- Проверь auth endpoints
- Найди hardcoded secrets
- Проверь input validation

ЭТАП 3: Качество кода
- Найди dead code
- Найди дублирование логики
- Проверь error handling

ЭТАП 4: Отчёт
- Сформируй таблицу проблем
- Приоритизируй по severity
- Предложи roadmap"
```

### 5.2 Prompt Chaining

Разбей на последовательные промпты:

```
Промпт 1: "Найди все endpoints в backend/api/routes/"
↓
Промпт 2: "Для каждого endpoint проверь наличие auth decorator"
↓
Промпт 3: "Для endpoints без auth — оцени риск"
↓
Промпт 4: "Сформируй отчёт с рекомендациями"
```

## 6. Структурирование промптов

### 6.1 XML-разметка (рекомендуется для Claude)

```xml
<task>
Проанализируй безопасность API endpoints
</task>

<context>
Проект: FastAPI backend
Аутентификация: JWT Bearer tokens
База: PostgreSQL через SQLAlchemy
</context>

<focus>
- SQL injection
- Authentication bypass
- Authorization flaws
</focus>

<output_format>
## Findings
| Endpoint | Vulnerability | Severity | Fix |
|----------|--------------|----------|-----|
</output_format>

<files>
{{вставь код}}
</files>
```

### 6.2 Markdown-секции

```markdown
# Задача
Рефакторинг модуля auth

# Контекст
- Python 3.10, FastAPI
- Текущая проблема: дублирование кода в 3 файлах
- Цель: единый AuthService

# Ограничения
- Сохранить обратную совместимость API
- Не менять схему БД
- Покрыть тестами

# Ожидаемый результат
1. Новый файл auth_service.py
2. Изменения в существующих файлах
3. Тесты
```

### 6.3 Нумерованные инструкции

```
Выполни ревью кода:

1. СНАЧАЛА прочитай весь файл
2. ЗАТЕМ найди все public методы
3. ДЛЯ КАЖДОГО метода проверь:
   3.1 Есть ли docstring
   3.2 Есть ли type hints
   3.3 Есть ли обработка ошибок
4. СОСТАВЬ таблицу с findings
5. ПРЕДЛОЖИ исправления для критичных проблем
```

## 7. Self-Consistency

### 7.1 Концепция

Генерируй несколько решений и выбирай консенсус.

```
Промпт:
"Предложи 3 разных способа рефакторинга этого кода.
Для каждого способа:
- Опиши подход
- Перечисли плюсы и минусы
- Оцени сложность (1-5)

Затем выбери лучший вариант и обоснуй."
```

### 7.2 Для debugging

```
Промпт:
"В коде есть баг. Сформулируй 3 гипотезы:

Гипотеза 1: [описание]
Как проверить: [шаги]
Вероятность: [%]

Гипотеза 2: ...
Гипотеза 3: ...

Какая гипотеза наиболее вероятна и почему?"
```

## 8. Итеративное уточнение

### 8.1 Паттерн уточнения

```
Промпт 1: "Найди проблемы в коде"
Ответ: [список проблем]

Промпт 2: "Раскрой подробнее проблему #3 —
покажи пример exploit и код исправления"
Ответ: [детали]

Промпт 3: "Проверь, что исправление не ломает
существующие тесты"
```

### 8.2 Refinement keywords

| Фраза | Эффект |
|-------|--------|
| "Раскрой подробнее..." | Углубляет ответ |
| "Приведи конкретный пример" | Добавляет примеры |
| "Упрости объяснение" | Делает проще |
| "А если [edge case]?" | Проверяет граничные случаи |
| "Какие ещё варианты?" | Расширяет ответ |
| "Сравни с [альтернативой]" | Добавляет сравнение |
