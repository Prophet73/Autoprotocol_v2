# Промпт: Полный аудит кодовой базы

> Пошаговое руководство для комплексного аудита проекта с использованием AI-ассистента

---

## Обзор процесса

```
┌─────────────────────────────────────────────────────────────┐
│                    ЭТАПЫ АУДИТА                             │
├─────────────────────────────────────────────────────────────┤
│  1. Структурный анализ     → Организация, мусор, дубликаты  │
│  2. Анализ backend         → Dead code, качество, паттерны  │
│  3. Анализ frontend        → Компоненты, стили, зависимости │
│  4. Security review        → Уязвимости, auth, валидация    │
│  5. Консолидация           → Отчёт, приоритизация, roadmap  │
└─────────────────────────────────────────────────────────────┘
```

## Этап 1: Структурный анализ

### Промпт 1.1: Исследование структуры

```
Исследуй структуру проекта. Найди:

1. ДУБЛИКАТЫ
   - Файлы с похожими именами (v1, v2, backup, old)
   - Папки с пересекающимся содержимым
   - Конфигурационные файлы в разных местах

2. МУСОР
   - Пустые или почти пустые файлы (< 10 строк)
   - Временные файлы (tmp, temp, .bak, ~)
   - Output/generated файлы в репозитории
   - Bundle/build артефакты

3. ORGANIZATION
   - Файлы в корне, которые должны быть в подпапках
   - Отсутствующие README в ключевых папках
   - Нарушения стандартной структуры проекта

Формат ответа:
## Дубликаты
| Файл 1 | Файл 2 | Тип дублирования |
|--------|--------|------------------|

## Мусор для удаления
| Файл/Папка | Размер | Причина удаления |
|------------|--------|------------------|

## Реорганизация
| Файл | Текущее место | Куда переместить |
|------|---------------|------------------|
```

### Промпт 1.2: Анализ .gitignore и secrets

```
Проанализируй .gitignore и найди:

1. СЕКРЕТЫ В РЕПОЗИТОРИИ
   - .env файлы (не .example)
   - Файлы с токенами, ключами, паролями
   - Конфигурации с credentials

2. НЕДОСТАЮЩИЕ ПАТТЕРНЫ
   - Какие типы файлов должны быть в .gitignore, но отсутствуют
   - IDE-специфичные файлы
   - OS-специфичные файлы

3. ЛИШНИЕ ПАТТЕРНЫ
   - Устаревшие или неиспользуемые исключения

Формат:
## Секреты для удаления из репо
| Файл | Содержит | Действие |
|------|----------|----------|

## Добавить в .gitignore
```gitignore
# [твои предложения]
```
```

## Этап 2: Анализ Backend

### Промпт 2.1: Dead code и unused imports

```
Проанализируй backend код на наличие:

1. НЕИСПОЛЬЗУЕМЫЕ ИМПОРТЫ
   - Импорты модулей, которые не используются
   - Импорты из удалённых/переименованных модулей

2. МЁРТВЫЙ КОД
   - Функции, которые нигде не вызываются
   - Классы без инстанцирования
   - Переменные, которые присваиваются, но не читаются
   - Недостижимый код после return/raise

3. ЗАКОММЕНТИРОВАННЫЙ КОД
   - Большие блоки закомментированного кода
   - TODO/FIXME которые никогда не будут сделаны

Для каждой находки укажи:
- Файл и строку
- Тип проблемы
- Безопасно ли удалить (да/нет/требует проверки)

Формат таблицы:
| Файл:строка | Тип | Код | Удалить? |
|-------------|-----|-----|----------|
```

### Промпт 2.2: Дублирование кода

```
Найди дублирование кода в backend:

1. COPY-PASTE КОД
   - Идентичные или почти идентичные блоки в разных файлах
   - Повторяющиеся паттерны (например, одинаковый error handling)

2. ДУБЛИРОВАНИЕ ЛОГИКИ
   - Одинаковая бизнес-логика в разных местах
   - Повторяющиеся валидации
   - Одинаковые SQL запросы

3. ВОЗМОЖНОСТИ ДЛЯ АБСТРАКЦИИ
   - Код, который можно вынести в utility функции
   - Повторяющиеся классы, которые можно обобщить

Формат:
## Дубликат #1: [название]
Места:
- `file1.py:10-25`
- `file2.py:45-60`

Код:
```python
[дублирующийся код]
```

Рекомендация:
[как устранить дублирование]
```

### Промпт 2.3: Качество и паттерны

```
Оцени качество backend кода:

1. ERROR HANDLING
   - Голые except без указания типа исключения
   - Игнорирование ошибок (pass в except)
   - Отсутствие обработки ошибок в критичных местах

2. ТИПИЗАЦИЯ
   - Функции без type hints
   - Неконсистентная типизация
   - Использование Any там, где можно указать конкретный тип

3. SOLID НАРУШЕНИЯ
   - Классы с множеством ответственностей
   - Жёсткие зависимости вместо DI
   - Нарушения принципа подстановки Лисков

4. ASYNC/AWAIT
   - Смешивание sync и async кода
   - Blocking calls в async функциях
   - Неправильное использование asyncio.run()

Формат:
## Проблема: [название]
**Severity:** Critical/High/Medium/Low
**Файл:** `path/to/file.py:123`
**Код:**
```python
[проблемный код]
```
**Проблема:** [описание]
**Исправление:**
```python
[исправленный код]
```
```

## Этап 3: Анализ Frontend

### Промпт 3.1: Компоненты и зависимости

```
Проанализируй frontend код:

1. НЕИСПОЛЬЗУЕМЫЕ КОМПОНЕНТЫ
   - Компоненты, которые не импортируются/рендерятся
   - Экспорты, которые нигде не используются

2. НЕИСПОЛЬЗУЕМЫЕ ЗАВИСИМОСТИ
   - Пакеты в package.json, которые не импортируются
   - devDependencies, которые можно удалить

3. ДУБЛИРОВАНИЕ
   - Компоненты с похожей функциональностью
   - Повторяющиеся стили (особенно inline)
   - Одинаковые константы в разных файлах

4. BARREL EXPORTS
   - index.ts файлы с неиспользуемыми экспортами
   - Circular dependencies

Формат:
## Неиспользуемые зависимости
| Пакет | Последнее использование | Действие |
|-------|------------------------|----------|

## Дублирование стилей
| Значение | Количество | Файлы | Решение |
|----------|------------|-------|---------|
```

### Промпт 3.2: Стили и Tailwind

```
Проанализируй использование стилей:

1. ХАРДКОД ЦВЕТОВ
   - Inline цвета вместо переменных/классов
   - Одинаковые цвета (#XXXXXX) в разных местах
   - Цвета, которые должны быть в theme

2. ДУБЛИРОВАНИЕ КЛАССОВ
   - Повторяющиеся комбинации Tailwind классов
   - Классы, которые можно вынести в @apply

3. НЕИСПОЛЬЗУЕМЫЕ СТИЛИ
   - CSS классы без использования
   - Tailwind классы в safelist без нужды

Формат:
## Захардкоженные цвета
| Цвет | Количество | Предложение |
|------|------------|-------------|
| #E52713 | 62 | Добавить в tailwind.config как 'brand' |

## Повторяющиеся комбинации
| Классы | Количество | Предложение |
|--------|------------|-------------|
```

## Этап 4: Security Review

### Промпт 4.1: OWASP Top 10

```
Ты — senior security engineer. Проверь код на OWASP Top 10:

1. INJECTION (A03:2021)
   - SQL injection
   - Command injection
   - LDAP injection

2. BROKEN AUTH (A07:2021)
   - Слабые пароли/токены
   - Session fixation
   - Missing rate limiting

3. SENSITIVE DATA EXPOSURE (A02:2021)
   - Hardcoded secrets
   - Logging sensitive data
   - Insecure transmission

4. XXE, SSRF, PATH TRAVERSAL
   - XML parsing без защиты
   - Unvalidated URLs
   - File path manipulation

5. SECURITY MISCONFIGURATION (A05:2021)
   - CORS too permissive
   - Debug mode in production
   - Default credentials

Для каждой находки:
| # | Vulnerability | CWE | File:Line | Severity | Exploit Scenario |
|---|--------------|-----|-----------|----------|------------------|

Исправление:
```python
# Было:
[уязвимый код]

# Стало:
[безопасный код]
```
```

### Промпт 4.2: Authentication & Authorization

```
Проверь систему аутентификации и авторизации:

1. AUTH ENDPOINTS
   - /login, /register, /logout
   - Password reset flow
   - Token refresh

2. JWT/SESSION
   - Алгоритм подписи (не none, не HS256 с коротким ключом)
   - Срок жизни токена
   - Refresh token rotation

3. AUTHORIZATION
   - RBAC/ABAC реализация
   - Privilege escalation возможности
   - IDOR (Insecure Direct Object Reference)

4. MIDDLEWARE
   - Auth middleware на всех protected routes
   - Корректная проверка ролей

Формат:
## Критичные проблемы
| Проблема | Файл | Impact | Fix |
|----------|------|--------|-----|
```

## Этап 5: Консолидация

### Промпт 5.1: Генерация отчёта

```
На основе всех найденных проблем создай консолидированный отчёт:

1. EXECUTIVE SUMMARY
   - Общее состояние проекта (1-10)
   - Критичные проблемы (количество)
   - Топ-3 приоритета

2. СТАТИСТИКА
   | Категория | Critical | High | Medium | Low |
   |-----------|----------|------|--------|-----|
   | Security  |          |      |        |     |
   | Code Quality |       |      |        |     |
   | Structure |          |      |        |     |

3. ДЕТАЛЬНЫЕ FINDINGS
   Для каждой проблемы:
   - ID
   - Категория
   - Severity
   - Файл:строка
   - Описание
   - Как исправить
   - Время на исправление

4. ROADMAP
   | Фаза | Задачи | Время | Приоритет |
   |------|--------|-------|-----------|
```

## Примеры запуска аудита

### Быстрый аудит (15 минут)

```
Проведи быстрый аудит проекта:
1. Найди критичные security issues
2. Найди очевидный мусор
3. Найди самые большие code smells

Лимит: топ-10 проблем, сортировка по severity.
```

### Полный аудит (1-2 часа)

```
Проведи полный аудит проекта по следующему плану:

ФАЗА 1 (параллельно):
- Агент 1: Структурный анализ
- Агент 2: Backend code quality
- Агент 3: Frontend analysis
- Агент 4: Security review

ФАЗА 2:
- Консолидация результатов
- Приоритизация
- Генерация roadmap

Формат итогового отчёта: Markdown с таблицами.
Сохранить в: docs/AUDIT_REPORT.md
```

### Целевой аудит (30 минут)

```
Проведи целевой аудит модуля аутентификации:

Scope:
- backend/core/auth/
- backend/api/routes/auth.py
- frontend/src/api/auth.ts
- frontend/src/contexts/AuthContext.tsx

Фокус:
- Security vulnerabilities
- Error handling
- Token management
- Session security

Игнорировать:
- Code style issues
- Missing documentation
- Minor refactoring opportunities
```

## Чеклист аудита

```markdown
## Pre-Audit
- [ ] Определён scope (весь проект / модуль / функция)
- [ ] Определены приоритеты (security first? quality first?)
- [ ] Есть доступ ко всем файлам
- [ ] Понятен технологический стек

## During Audit
- [ ] Структурный анализ завершён
- [ ] Backend проанализирован
- [ ] Frontend проанализирован
- [ ] Security review проведён
- [ ] Все findings задокументированы

## Post-Audit
- [ ] Findings приоритизированы
- [ ] Roadmap создан
- [ ] Отчёт сохранён
- [ ] Critical issues выделены
```
