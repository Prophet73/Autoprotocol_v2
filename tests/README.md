# Тестирование SeverinAutoprotocol

В проекте используется прагматичный подход к тестированию. Так как проект разрабатывается небольшой командой/соло, мы не пишем юнит-тесты на каждый метод, а фокусируемся на **интеграционных тестах** критических путей.

## Структура
- `tests/backend/` - тесты API и утилит (FastAPI + pytest)
- `tests/frontend/` - (в планах) E2E тесты критического пути через Playwright

## Запуск тестов (Backend)

Тесты используют легковесную SQLite in-memory БД и моки для тяжелых ML-моделей (PyTorch, Whisper, Pyannote не загружаются в память).

```bash
# 1. Активировать виртуальное окружение
source venv310/bin/activate

# 2. Установить тестовые зависимости (если еще не)
pip install pytest pytest-asyncio httpx aiosqlite

# 3. Запустить все тесты
pytest tests/ -v

# Запуск конкретного файла
pytest tests/backend/test_job_management.py -v
```

## Что обязательно нужно тестировать при доработках

- **Парсеры LLM**: Если меняете `backend/domains/*/schemas.py`, проверьте, что старые JSON-ответы всё ещё парсятся.
- **Безопасность**: Файловая система (`test_file_security.py`) и защита инъекций.
- **Состояния Celery**: Отмена задач, статусы.

## Что мы НЕ тестируем (Mock-ируется)

- Работу самого WhisperX и Pyannote (тестируется вручную на реальном GPU).
- Соединение с реальным Gemini API в автоматических пайплайнах (чтобы не тратить токены).
