## 1. Принципы: что изменилось с Structured Output

### Старый подход (до structured output)

```
Промпт → Gemini → свободный текст → парсинг JSON → валидация → ошибки
```

Промпты были нагружены инструкциями формата: "верни JSON с полями...", "обязательно используй
ключи...", "формат ответа должен быть...". Большая часть токенов тратилась на описание структуры
вместо содержательных инструкций.

### Новый подход (structured output)

```
Промпт + JSON Schema → Gemini → гарантированный валидный JSON → Pydantic parse → готово
```

Вызов LLM:

```python
response = client.models.generate_content(
    model=REPORT_MODEL,
    contents=full_prompt,
    config=types.GenerateContentConfig(
        system_instruction=system_prompt,
        response_mime_type="application/json",
        response_json_schema=ResultModel.model_json_schema(),
    ),
)
result = ResultModel.model_validate_json(response.text)
```

Gemini **видит** JSON Schema (включая `description` из `Field(...)`) и **гарантирует** валидный
JSON. Это значит:

| Раньше приходилось делать в промпте | Теперь делает schema автоматически |
|---|---|
| "Верни JSON" | `response_mime_type="application/json"` |
| "Используй поля: title, description, ..." | `model_json_schema()` передает все поля |
| "Тип поля status должен быть строкой" | `type: string` в schema |
| "Допустимые значения: high/medium/low" | `enum: [high, medium, low]` в schema |
| "Поле может быть null" | `anyOf: [{type: string}, {type: null}]` в schema |
| "description — это что конкретно нужно сделать" | `Field(description="Что конкретно нужно сделать")` |

### Главный вывод

**Промпт должен объяснять ЧТО извлекать и КАК рассуждать, а не КАК форматировать.**

Бюджет токенов промпта теперь надо тратить на:
- Роль и контекст экспертизы
- Правила извлечения информации (когда писать null, как оценивать уверенность)
- Антигаллюцинационные инструкции
- Интроспекцию ("подумай, прежде чем заполнять")
- Калибровочные мини-примеры

## 2. Что убрать из промптов

### 2.1. Инструкции о формате JSON

**Убрать** все блоки вида "верни JSON", "структура ответа", "ПРАВИЛА И СТРУКТУРА (JSON)".

Пример текущего промпта (DCT brainstorm):

```yaml
# ПРАВИЛА И СТРУКТУРА (JSON)
1. **session_topic**: Тема, которой был посвящен штурм.
2. **main_problem**: Какую проблему или задачу решали?
3. **idea_clusters**: Список кластеров идей. Каждый кластер — ...
4. **top_ideas**: Список из 3-5 самых перспективных идей. Каждая ...
```

Эта секция дублирует `BrainstormResult.model_json_schema()`, который уже содержит:
- `session_topic` с `description: "Тема сессии"`
- `main_problem` с `description: "Основная проблема/задача"`
- `idea_clusters` с вложенной schema `BrainstormIdeaCluster`
- и т.д.

**Исключение:** Если нужно передать содержательную семантику, которую невозможно выразить
через `Field(description=...)`, например:

```
idea_clusters: сгруппируй идеи ПО НАПРАВЛЕНИЯМ (маркетинг, продукт, технологии),
а не по хронологии обсуждения
```

Такое *как именно группировать* — это содержательная инструкция, ее оставляем в промпте.
Но "Каждый кластер — объект с полями cluster_name и ideas" — убираем, это есть в schema.

### 2.2. Описания полей, совпадающие со schema

Если в промпте написано:

```
- responsible: кто отвечает (ФИО или организация), null если не указано
```

А в Pydantic:

```python
responsible: Optional[str] = Field(None, description="Ответственный (ФИО или организация)")
```

То промпт дублирует schema. Убираем из промпта, оставляем в `Field(description=...)`.

### 2.3. `<output_format>` секции

В risk_brief промпте есть `<output_format>` блок, дублирующий schema:

```xml
<output_format>
Верни JSON согласно схеме RiskBrief:
- project_name, project_code, location (null если не упоминается)
- overall_status: stable/attention/critical
...
</output_format>
```

Убираем полностью. Schema это покрывает.

### 2.4. Инструкции "для каждого укажи поля X, Y, Z"

Текущий промпт construction basic_report:

```
## Для каждой задачи укажи:
- category: категория из списка выше (точное значение)
- description: что конкретно нужно сделать
- responsible: кто отвечает, null если не указано
- deadline: срок, null если не указан
...
```

Это полностью дублирует `Task.model_json_schema()`. Убираем.

Оставляем только **содержательные** инструкции: правила извлечения, уровни уверенности,
принцип "проблема = задача".

## 3. Что добавить в промпты

### 3.1. Интроспекция (Think Before Output)

Принцип: заставить модель порассуждать перед заполнением полей.

В system prompt добавляем:

```yaml
system: |
  МЕТОД РАБОТЫ:
  Прежде чем формировать JSON, мысленно пройди три шага:
  1. СКАНИРОВАНИЕ — прочитай стенограмму целиком, отметь ключевые факты
  2. ВЕРИФИКАЦИЯ — для каждого факта проверь: это сказано явно или ты додумываешь?
  3. ЗАПОЛНЕНИЕ — только после верификации заполняй поля JSON
```

Для risk_brief (где уже есть INoT подход) — усилить верификацию:

```yaml
system: |
  Перед финальным JSON выполни внутренний аудит:
  - Каждый риск имеет evidence (цитату)? Если нет — перенеси в hypotheses.
  - Каждая mitigation содержит КТО + ЧТО + КОГДА? Если нет — переформулируй.
  - decision заполнен ТОЛЬКО фактами из стенограммы? Если это твой вывод — перенеси в mitigation.
```

### 3.2. Маркеры качества

Добавить в system prompt критерии самооценки:

```yaml
system: |
  КРИТЕРИИ КАЧЕСТВА:
  - Нет выдуманных фактов (каждое утверждение подкреплено текстом)
  - Нет пустых формулировок ("необходимо усилить контроль" — запрещено)
  - Конкретные имена, даты, цифры из стенограммы
  - Если данных недостаточно — честный null вместо домысла
```

### 3.3. Evidence-gating (доказательная база)

Принцип: **нет упоминания в тексте = null**.

```yaml
system: |
  EVIDENCE-GATING:
  Для полей responsible, deadline, evidence действует правило:
  Если факт НЕ УПОМЯНУТ ЯВНО в стенограмме — поле ДОЛЖНО быть null.
  Запрещено:
  - Выводить ответственного из контекста ("наверное, это Иванов")
  - Придумывать сроки ("вероятно, до конца месяца")
  - Генерировать цитаты, которых нет в тексте
```

### 3.4. NULL-FIRST принцип

```yaml
system: |
  NULL-FIRST:
  При заполнении Optional-полей начинай с предположения null.
  Заполняй значением ТОЛЬКО если нашёл явное подтверждение в тексте.
  Лучше честный null, чем красивая выдумка.
```

### 3.5. Контекст даты

Добавить в user prompt якорь для временных расчетов:

```yaml
user: |
  Дата совещания: {meeting_date}
  Сегодняшний день для расчёта сроков: {meeting_date}
  "На следующей неделе" = неделя после {meeting_date}.
  "В пятницу" = ближайшая пятница после {meeting_date}.
```

### 3.6. Принцип SRP (Single Responsibility Prompt)

Каждый промпт отвечает за одну задачу. Не смешивать в одном промпте:
- Классификацию типа встречи + извлечение задач + экспертный анализ (как сейчас в basic_report)

Если смешивание необходимо (один LLM-вызов из соображений стоимости/скорости), то четко
разделять блоки в промпте через нумерацию этапов.

## 4. Структура промптов: System vs User

### System prompt (постоянная часть)

Содержит:
- **Роль** модели (кто ты, какая экспертиза)
- **Язык** вывода (всегда русский)
- **Константные правила** (NULL-FIRST, evidence-gating, антигаллюцинации)
- **Метод работы** (интроспекция, этапы рассуждения)
- **Критерии качества**
- **Запреты** (что нельзя делать)

Не содержит:
- Стенограмму (она переменная)
- Дату совещания (она переменная)
- Описание формата JSON (это в schema)

### User prompt (переменная часть)

Содержит:
- **Дату совещания** с якорем для расчёта сроков
- **Тип задачи** (что именно нужно сделать с этой стенограммой)
- **Содержательные подсказки** для конкретного типа встречи
- **Стенограмму** (в конце, чтобы модель сначала прочитала инструкции)

### Шаблон

```yaml
domains:
  <domain>:
    <meeting_type>:
      system: |
        <Role>
        Ты — [роль]. [1-2 предложения о специализации].
        </Role>

        <Language>
        Весь вывод строго на русском языке.
        </Language>

        <Method>
        1. СКАНИРОВАНИЕ: прочитай стенограмму целиком
        2. ВЕРИФИКАЦИЯ: проверь каждый факт — он явный или додуманный?
        3. ЗАПОЛНЕНИЕ: только подтверждённые факты → в поля JSON
        </Method>

        <Rules>
        - NULL-FIRST: Optional-поле = null, пока нет явного подтверждения
        - EVIDENCE-GATING: нет упоминания в тексте = null
        - БЕЗ ВЫДУМОК: запрещено генерировать факты, имена, даты
        - КОНКРЕТИКА: вместо "усилить контроль" → "[Кто] проверит [что] до [когда]"
        </Rules>

        <Quality>
        Перед формированием ответа проверь:
        - Каждый факт можно найти в стенограмме?
        - Нет ли "красивых" но выдуманных формулировок?
        - Все Optional-поля без подтверждения = null?
        </Quality>

      user: |
        Дата совещания: {meeting_date}

        [Содержательные инструкции для конкретного типа — ЧТО извлекать, КАК оценивать]

        Стенограмма:
        ---
        {transcript}
        ---
```

### Разделение system/user в коде

Текущий код risk_brief уже правильно разделяет:

```python
config=types.GenerateContentConfig(
    system_instruction=RISK_BRIEF_SYSTEM,   # ← system prompt
    ...
)
contents=user_prompt,  # ← user prompt
```

А basic_report и analysis передают оба через `contents`:

```python
contents=[system_prompt, user_prompt] if system_prompt else user_prompt,
```

Рекомендация: **всегда** использовать `system_instruction` для system prompt.
Это семантически правильнее и дает модели четкий контекст.

## 5. Field(description=...) в Pydantic моделях

### Почему это важно

Когда мы вызываем `ResultModel.model_json_schema()`, все `description` из `Field(...)` попадают
в JSON Schema, которую видит Gemini. Фактически **Field(description=...) — это часть промпта**.

### Текущее состояние: слабые description

Многие description в проекте слишком общие:

```python
# Плохо — общие слова
responsible: Optional[str] = Field(None, description="Ответственный")
deadline: Optional[str] = Field(None, description="Срок")
status: str = Field(..., description="Статус")
```

Gemini видит `"description": "Ответственный"` и не знает:
- В каком формате писать (ФИО? Организация? "Иванов И.И."?)
- Когда ставить null
- Что делать, если ответственный не назван явно

### Формат хорошего Field(description)

```
[Что это] + [формат/примеры] + [null-политика] + [антипаттерн]
```

### Примеры трансформации

**responsible**:

```python
# Было:
responsible: Optional[str] = Field(None, description="Ответственный")

# Стало:
responsible: Optional[str] = Field(
    None,
    description=(
        "ФИО или название организации, ответственной за выполнение. "
        "Только если явно назначен в стенограмме. "
        "null если ответственный не назван."
    )
)
```

**deadline**:

```python
# Было:
deadline: Optional[str] = Field(None, description="Срок")

# Стало:
deadline: Optional[str] = Field(
    None,
    description=(
        "Срок выполнения в формате ДД.ММ.ГГГГ. "
        "Относительные даты ('на следующей неделе') пересчитывать от даты совещания. "
        "null если срок не озвучен."
    )
)
```

**evidence**:

```python
# Было:
evidence: Optional[str] = Field(
    None, description="Краткая цитата из стенограммы, подтверждающая задачу"
)

# Стало:
evidence: Optional[str] = Field(
    None,
    description=(
        "Дословная краткая цитата из стенограммы (1-2 предложения), "
        "подтверждающая существование задачи. "
        "Запрещено перефразировать или придумывать цитату. "
        "null если явного упоминания нет."
    )
)
```

**meeting_summary**:

```python
# Было:
meeting_summary: str = Field(
    description="О чём говорили — 2-3 предложения, суть совещания"
)

# Стало:
meeting_summary: str = Field(
    description=(
        "Суть совещания в 2-3 предложениях: основные темы, ключевые решения, "
        "общий итог. Только факты из стенограммы, без оценок."
    )
)
```

**status поля с enum**:

```python
# Было:
status: str = Field(..., description="Статус: выполнено / в работе / задержка / заблокировано")

# Стало (с enum):
class TaskStatusEnum(str, Enum):
    DONE = "Выполнено"
    IN_PROGRESS = "В работе"
    DELAYED = "Задержка"
    BLOCKED = "Заблокировано"

status: TaskStatusEnum = Field(
    ...,
    description=(
        "Текущий статус задачи на основании обсуждения. "
        "Выполнено — только если явно подтверждено завершение."
    )
)
```

### Правила для Field(description)

1. **Пиши на русском** — Gemini видит description и генерирует на этом языке.
2. **Указывай формат** — "в формате ДД.ММ.ГГГГ", "ФИО или организация", "1-2 предложения".
3. **Указывай null-политику** — "null если не упомянут", "null если нет явного факта".
4. **Добавляй антипаттерн** — "Запрещено перефразировать", "Не выдумывать цитату".
5. **Ограничивай длину** — description не должен быть длиннее 2-3 предложений.
6. **Не дублируй type** — не пиши "строка" если поле и так `str`.
7. **Используй Enum вместо строк** — если есть фиксированный набор значений, Enum попадает
   в schema как `enum: [...]` и Gemini не сможет выбрать другое значение.

## 6. Few-shot: мини-примеры для калибровки

### Зачем few-shot при structured output

Schema гарантирует формат, но НЕ гарантирует качество содержания. Few-shot нужен чтобы:
- Откалибровать уровень детализации (насколько подробно описывать)
- Показать правильный стиль формулировок
- Продемонстрировать null-политику в действии
- Задать тон (формальный vs краткий)

### Формат: калибровочный мини-пример в system prompt

Не нужно давать полный пример ответа (schema это покрывает). Достаточно 1-2 фрагмента:

```yaml
system: |
  ...

  <Calibration>
  Пример хорошей задачи:
  {
    "description": "Направить акт скрытых работ по устройству фундамента блока Б в экспертизу",
    "responsible": "Петров А.В.",
    "deadline": "15.02.2026",
    "priority": "high",
    "confidence": "high",
    "evidence": "Петров, акт по фундаменту блока Б давайте до пятницы в экспертизу отправим"
  }

  Пример плохой задачи (НЕ ДЕЛАЙ ТАК):
  {
    "description": "Разобраться с документами",
    "responsible": "кто-то из отдела",
    "deadline": "скоро",
    "priority": "medium",
    "confidence": "high",
    "evidence": null
  }
  Почему плохо: неконкретная формулировка, нет ФИО, нет даты, confidence=high без evidence.
  </Calibration>
```

### Few-shot для разных задач

**Для risk_brief — калибровка severity**:

```yaml
<Calibration>
Пример evidence-based risk (ПРАВИЛЬНО):
- title: "Задержка поставки металлоконструкций"
- evidence: "поставщик сказал, что задержка минимум 2 недели из-за логистики"
- probability: 4, impact: 3, confidence: high

Пример hallucinated risk (НЕПРАВИЛЬНО):
- title: "Возможные проблемы с качеством бетона"
- evidence: null (в тексте бетон не обсуждался)
- Такой риск → hypotheses, не risks
</Calibration>
```

**Для переговоров — калибровка positions**:

```yaml
<Calibration>
Хорошо: "Подрядчик настаивает на авансе 30%, заказчик предложил 15% с промежуточными актами"
Плохо: "Стороны обсудили финансовые условия" (слишком размыто)
</Calibration>
```

### Ограничения

- Не более 2 калибровочных примеров на промпт — иначе модель начнет копировать содержание.
- Few-shot в **system** prompt (не в user) — чтобы не смешивать с данными стенограммы.
- Примеры должны быть **из другой** тематики, чтобы модель не скопировала конкретные имена/факты.

## 7. Мультиязычность

### Проблема

Стенограммы могут содержать реплики на разных языках (китайский, английский, турецкий) —
когда на совещании присутствуют иностранные подрядчики. WhisperX транскрибирует все языки
в одну стенограмму.

### Текущий подход (правильный, нужно усилить)

В промптах уже есть правила мультиязычности:

```
МУЛЬТИЯЗЫЧНОСТЬ:
Если в речи есть иностранные языки (китайский, английский), НЕ ПЕРЕВОДИ их дословно.
Извлекай СУТЬ и пиши отчет СТРОГО НА РУССКОМ ЯЗЫКЕ.
```

### Улучшенный подход

Перенести базовое правило мультиязычности в **system prompt** (оно константное для всех запросов):

```yaml
system: |
  <Language>
  Весь вывод строго на русском языке.
  Стенограмма может содержать реплики на разных языках (китайский, английский, турецкий).
  Правила:
  - НЕ переводи дословно — извлекай СУТЬ и формулируй на русском
  - Технические термины оставляй в оригинале если нет устоявшегося перевода
  - Имена собственные транслитерируй: "Zhang Wei" → "Чжан Вэй"
  - Названия компаний оставляй в оригинале: "CSCEC", "Samsung C&T"
  - При извлечении фактов из иностранной части добавляй пометку:
    [из китайской/английской части переговоров]
  </Language>
```

### Специфика для разных типов встреч

**Переговоры** — часто одна сторона говорит на одном языке, другая на другом:

```yaml
# В user prompt для negotiation:
Если переговоры велись на нескольких языках, резюмируй позицию каждой стороны
отдельно, указывая язык: [из англоязычной части], [из китайской части].
```

**Лекция** — лектор может говорить на иностранном языке:

```yaml
# В user prompt для lecture:
Если лекция на иностранном языке, создай конспект на русском.
Не переводи дословно — передавай смысл и структуру знаний.
Ключевые термины давай с оригиналом в скобках: "машинное обучение (machine learning)".
```

## 8. Антигаллюцинации

### Типичные галлюцинации в нашем проекте

1. **Придуманные ответственные** — модель назначает людей, которых нет в стенограмме
2. **Придуманные сроки** — "до конца недели" когда в тексте срок не обсуждался
3. **Придуманные цитаты** — evidence содержит текст, которого нет в стенограмме
4. **Раздувание рисков** — модель ставит probability=5 impact=5 на мелкий вопрос
5. **Генерация "красивых" формулировок** — "Необходимо провести комплексный аудит" вместо null

### Антигаллюцинационный блок (добавить в каждый system prompt)

```yaml
<AntiHallucination>
ЗАПРЕЩЕНО:
1. Придумывать имена, организации, даты, которых нет в тексте
2. Генерировать evidence/цитаты — только дословные фрагменты из стенограммы
3. Заполнять Optional-поля "для красоты" — если нет факта → null
4. Выдумывать mitigation вида "необходимо усилить контроль" (пустышка)
5. Ставить high confidence без evidence
6. Приписывать решения, которые не были приняты на совещании

ВМЕСТО ЭТОГО:
1. null + честный confidence
2. "Ответственный не назначен" (в description задачи), null (в поле responsible)
3. Перенос в hypotheses при отсутствии evidence
</AntiHallucination>
```

### Evidence-gating: механизм

Для каждого поля с доказательной базой (evidence, responsible, deadline, decision) действует
правило трех вопросов:

```
1. Это написано ДОСЛОВНО в стенограмме?
   → Да: заполняй
   → Нет: переходи к вопросу 2

2. Это ОДНОЗНАЧНО следует из контекста?
   → Да: заполняй + confidence=medium + пометка [выведено из контекста]
   → Нет: переходи к вопросу 3

3. Это мой ДОМЫСЕЛ?
   → null
```

### NULL-FIRST в Pydantic

Усилить через Field(description):

```python
responsible: Optional[str] = Field(
    None,
    description=(
        "ФИО или организация. ТОЛЬКО из стенограммы. "
        "null по умолчанию, заполнять только при явном назначении."
    )
)
```

### Hallucination patterns (уже есть в config)

В `prompts.yaml` есть `hallucination_patterns` — регулярки для фильтрации артефактов WhisperX.
Это пост-обработка, не относится к промптам. Но полезно упомянуть:

```yaml
hallucination_patterns:
  - "спасибо за (просмотр|внимание|подписку)"
  - "подписывайтесь на канал"
  - ...
```

Эти паттерны фильтруются ДО отправки в LLM, на этапе транскрипции.

## 9. По типам встреч

### 9.1. Производственное совещание (construction basic_report, dct production)

**Специфика:** Четкие поручения, сроки, ответственные. Много конкретики.

**Техники:**
- Максимальное извлечение задач (лучше больше для ручной фильтрации)
- Проблема = задача (каждая озвученная проблема порождает задачу)
- Дедупликация (одна задача из нескольких упоминаний)
- Расчет дат от {meeting_date}

**Ключевой принцип для промпта:**
```
ИЗВЛЕЧЕНИЕ > ФИЛЬТРАЦИЯ
Лучше извлечь 20 задач и дать менеджеру отфильтровать 5 нерелевантных,
чем упустить 3 критические.
```

### 9.2. Переговоры (negotiation)

**Специфика:** Позиции сторон, скрытые мотивы, условия, риски.

**Техники:**
- Разделение позиций: четко разграничить "наша сторона предложила" vs "контрагент ответил"
- Скрытые мотивы: анализировать не только слова, но логику аргументации
- Условия: точные цифры, проценты, сроки из текста
- Разделение на "согласовано" / "отложено" / "отклонено"

**Ключевой принцип для промпта:**
```
Резюмируй ПОЗИЦИИ, а не хронологию.
Плохо: "Сначала обсудили цену, потом сроки, потом гарантии"
Хорошо: "По цене: подрядчик запросил 15М, заказчик ограничил 12М, сошлись на 13.5М"
```

### 9.3. Мозговой штурм (brainstorm)

**Специфика:** Идеи без критики, кластеризация, приоритизация.

**Техники:**
- Зафиксировать ВСЕ идеи (даже "безумные") — потом команда отфильтрует
- Группировать по направлениям, не по хронологии
- Не оценивать качество идей — только фиксировать impact/complexity
- Parked ideas — идеи, которые явно отложили ("это потом", "не сейчас")

**Ключевой принцип для промпта:**
```
ФИКСАЦИЯ > ОЦЕНКА
Не критикуй идеи. Не фильтруй "нереалистичные". Зафиксируй всё.
Оценка impact/complexity — только для top_ideas.
```

### 9.4. Лекция / Вебинар (lecture)

**Специфика:** Один спикер, структурированная подача, Q&A секция.

**Техники:**
- Блочная структура (лектор обычно идет по плану)
- Тайм-коды к каждому блоку
- Тезисы — суть идеи, а не пересказ слов
- Q&A — вопрос + ключевая мысль ответа

**Ключевой принцип для промпта:**
```
КОНСПЕКТ > СТЕНОГРАММА
Не пересказывай слова лектора. Извлекай структуру знаний.
Каждый блок = заголовок + ключевая мысль + тезисы.
```

### 9.5. Risk Brief (construction risk_brief)

**Специфика:** Самый сложный тип. Executive-отчет с матрицей рисков, INoT-подход.

**Техники:**
- Мульти-агентный дебат (Agent_Risk vs Agent_Skeptic)
- Фазовый анализ с верификацией
- Строгая калибровка probability/impact (не ставить 5/5 без оснований)
- Decision vs Mitigation разделение
- Drivers для критических рисков (score >= 9)

**Ключевой принцип для промпта:**
```
ФАКТЫ > ИНТЕРПРЕТАЦИЯ
Извлекай только то, что ЯВНО сказано. Каждый риск = evidence из текста.
Нет evidence → нет риска (hypotheses).
probability=5 ТОЛЬКО если "точно будет", не "возможно".
```

### 9.6. Стратегическое планирование (strategic_planning)

**Специфика:** Долгосрочные цели, инициативы, KPI, абстрактные обсуждения.

**Техники:**
- Разграничение целей (что хотим) и инициатив (как достигнем)
- KPI с конкретными таргетами
- Риски — стратегические, а не операционные

**Ключевой принцип для промпта:**
```
КОНКРЕТИЗАЦИЯ
Стратегическая сессия часто содержит размытые формулировки.
Задача: превратить "нужно расти" в конкретную инициативу с ответственным.
Если конкретики нет в тексте — честно это отразить, а не придумывать.
```

### 9.7. Встреча с клиентом (client_meeting)

**Специфика:** Потребности клиента, предложенные решения, возражения.

**Техники:**
- Разделить потребности клиента (что хочет) от предложенных решений (что мы предлагаем)
- Зафиксировать возражения и реакции (client_feedback)
- Четкие next_steps с разделением "мы делаем" / "клиент делает"

**Ключевой принцип для промпта:**
```
КЛИЕНТ > МЫ
Приоритет — зафиксировать потребности и обратную связь клиента.
Наши предложения — вторичны, если клиент их не подтвердил.
```

### 9.8. Рабочее совещание (work_meeting)

**Специфика:** Статусы, блокеры, решения, новые задачи.

**Техники:**
- Четкие статусы задач (Выполнено/В работе/Задержка/Заблокировано)
- Блокеры выделять отдельно
- Решения — только принятые на совещании, не предложения

**Ключевой принцип для промпта:**
```
СТАТУС > ОБСУЖДЕНИЕ
Фиксируй конечный статус, а не процесс обсуждения.
"Обсудили задачу X" — бесполезно. "Задача X: в работе, дедлайн сдвинут на 20.02" — полезно.
```

### 10.4. Пример трансформации: DCT brainstorm

**БЫЛО (текущий промпт):**

```yaml
brainstorm:
  system: |
    Ты — креативный фасилитатор мозгового штурма. Твоя задача — зафиксировать все идеи
    без критики, сгруппировать их и помочь команде выбрать самые перспективные.

  user: |
    Дата проведения: {meeting_date}

    Проанализируй запись сессии мозгового штурма. Создай структурированный отчет.

    # ПРАВИЛА И СТРУКТУРА (JSON)
    0. **ЯЗЫК**: Если идеи озвучивались на иностранном языке, сформулируй их суть
       на русском. В скобках можно оставить оригинальное название идеи, если это важно.
    1. **session_topic**: Тема, которой был посвящен штурм.
    2. **main_problem**: Какую проблему или задачу решали?
    3. **idea_clusters**: Список кластеров идей. Каждый кластер — это объект с полями
       `cluster_name` (направление, например 'Маркетинг') и `ideas` (список всех идей
       по этому направлению).
```

**БЫЛО (текущий system):**

```yaml
system: |
```

**СТАЛО (улучшенный system):**

## Приложение А: Как schema передается в Gemini

При вызове:

```python
config=types.GenerateContentConfig(
    response_json_schema=Task.model_json_schema(),
)
```

Gemini получает JSON Schema примерно такого вида:

```json
{
  "type": "object",
  "properties": {
    "category": {
      "enum": ["ИРД", "Проектирование и РД", "СМР", ...],
      "description": "Категория задачи",
      "type": "string"
    },
    "description": {
      "description": "Что нужно сделать",
      "type": "string"
    },
    "responsible": {
      "anyOf": [{"type": "string"}, {"type": "null"}],
      "default": null,
      "description": "Ответственный (ФИО или организация)"
    },
    "deadline": {
      "anyOf": [{"type": "string"}, {"type": "null"}],
      "default": null,
      "description": "Срок выполнения"
    },
    "confidence": {
      "enum": ["high", "medium"],
      "default": "high",
      "description": "Уровень уверенности: high=явная, medium=из контекста"
    },
    "evidence": {
      "anyOf": [{"type": "string"}, {"type": "null"}],
      "default": null,
      "description": "Краткая цитата из стенограммы, подтверждающая задачу"
    }
  },
  "required": ["category", "description"]
}
```

Модель видит ВСЕ эти description и использует их как инструкции при заполнении.
Поэтому `Field(description=...)` — это не просто документация, а часть промпта.

## Приложение Б: Порядок приоритетов инструкций

Когда Gemini получает конфликтующие инструкции, приоритет (от высшего к низшему):

1. **JSON Schema constraints** (enum, type, required) — безусловное ограничение
2. **system_instruction** — роль, правила, метод
3. **Field(description)** в schema — семантика полей
4. **user prompt** — конкретная задача и данные

Если в system написано "responsible = null если не назначен", а в user промпте было бы "заполни
все поля" — system побеждает. Поэтому антигаллюцинационные правила ставим в system.

## Приложение В: Быстрая проверка нового промпта

Перед деплоем нового промпта, проверь по чеклисту:

- [ ] Нет описаний полей JSON, дублирующих schema
- [ ] Есть блок `<Method>` с этапами рассуждения
- [ ] Есть блок `<AntiHallucination>` с NULL-FIRST
- [ ] Есть `<Language>` с правилом "вывод на русском"
- [ ] Есть `<Calibration>` с 1-2 мини-примерами (good/bad)
- [ ] System содержит только константные правила (без данных)
- [ ] User содержит дату, задачу и стенограмму (в конце)
- [ ] Используется `system_instruction=` в коде (не `contents=[]`)
- [ ] Field(description) содержит формат + null-политику + антипаттерн
- [ ] Enum используется вместо строковых ограничений где возможно
